# 昇腾 NPU 与 NVIDIA GPU 的设计思想：从第一性原理出发

## 3. 昇腾 NPU：面向张量数据流的专用加速器

### 3.1 目标工作负载

昇腾 NPU 的主要目标是以较高能效执行规则张量计算。

神经网络中的主要计算通常可表示为矩阵乘、卷积或向量后处理。卷积经过 tiling、数据重排或专用布局后，也会表现为规则的乘加阵列计算。这类工作负载具有以下性质：

- 算子类型集中，矩阵乘和卷积占主要计算量。
- 数据访问模式较规则，许多依赖关系可在编译期分析。
- 算术密度高，但片外带宽仍可能成为限制因素。
- 性能高度依赖 tile 划分、片上缓存复用和搬运计算重叠。

因此，NPU 的合理设计目标是围绕张量 tile 构造可预测的数据流执行管线，而不是提供最通用的线程执行环境。

### 3.2 从矩阵乘推导架构需求

考虑矩阵乘：

```text
C = A × B
```

每个输出元素需要复用 A 的一行和 B 的一列。如果每次乘加都从片外读取操作数，带宽需求会迅速超过可承受范围。因此，架构必须将矩阵分块，并在片上反复复用当前 tile。

由此可推导出 NPU 的基本需求：

1. 片外或共享存储保存完整张量。
2. 片上缓冲保存当前工作 tile。
3. 专用矩阵阵列持续消费 L0/L1 附近的数据。
4. 数据搬运单元提前搬入后续 tile，并搬出已完成结果。
5. 同步机制在 producer 和 consumer 之间建立明确依赖边界。

昇腾式 AI Core 中的 Cube、Vector、Scalar、MTE 和多级片上存储，正是对这些需求的结构化实现。

### 3.3 Cube 的作用

Cube 单元是面向矩阵乘的专用执行部件。它通常以固定 tile 为操作粒度，从 L0A/L0B 读取矩阵块，并将结果写入 L0C。

这种专用化带来三个直接收益：

- 阵列结构规则，面积和能耗效率较高。
- 数据通路固定，便于构建高吞吐流水。
- 编译器可围绕 tile、buffer 和 event 进行静态或半静态调度。

Cube 不能孤立发挥作用。专用矩阵阵列提供的是峰值乘加能力；该能力能否转化为有效吞吐，取决于片上数据能否按 tile 持续供给。如果数据供应不足，大量 PE 会因等待操作数而空闲。因此，昇腾 NPU 并不是先孤立设计矩阵阵列、再事后补充数据搬运机制，而是围绕规则张量计算同时设计计算阵列、片上存储、MTE 搬运和同步机制。换言之，矩阵阵列决定峰值计算能力，显式数据流体系决定该峰值能否被持续接近。

### 3.4 MTE 与显式数据流

在上述约束下，昇腾式 NPU 将数据搬运建模为独立受调度资源，而不是计算指令的附属动作。

MTE 负责不同存储层次之间的数据移动，例如全局存储到 UB、UB 到 L0A/L0B、L0C 到 UB。软件或编译器需要显式组织以下流水：

```text
CopyIn  ->  Compute  ->  CopyOut
```

这不同于主要依赖 cache 自动缓解访存延迟的通用处理器路线。NPU 假定程序能够以 tile 为单位预测数据需求，因此将更多调度责任交给编译器、运行时和显式同步机制：

- 哪个 tile 应先搬入。
- 哪段 UB 可被复用。
- 哪个 event 表示生产者已经完成。
- 哪个计算可与哪次搬运并行。

这种设计提高了数据流的确定性和能效，但要求编程模型、编译器和运行时承担更复杂的调度责任。

### 3.5 设计思想总结

昇腾 NPU 的体系结构思想可概括为：

```text
用专用张量阵列提高单位能耗算力；
用显式片上存储层次提高数据复用；
用 MTE 和 event/token 将数据搬运组织为可预测流水。
```

它优先优化规则张量计算的能效和确定性吞吐，而不是任意控制流程序的通用执行能力。

## 4. NVIDIA GPU：面向大规模线程并行的通用吞吐处理器

### 4.1 目标工作负载

NVIDIA GPU 的初始目标是实时图形渲染。图形渲染与神经网络张量计算不同：它需要在固定帧时间内处理大量顶点、三角形、片元和像素。各元素通常执行相似程序，但输入数据、访存地址、控制流路径和输出行为可能不同。

例如，fragment shader 中的大量片元通常执行同一程序框架：读取材质参数，采样纹理，计算法线和光照，输出颜色。但不同片元具有不同的屏幕坐标、纹理坐标、深度值、法线和材质属性；部分片元可能被深度测试丢弃，部分片元可能执行透明混合，部分片元可能因阴影、法线贴图或条件分支进入不同路径。顶点处理也具有类似性质：所有顶点通常经过坐标变换和属性准备，但不同顶点的模型矩阵、骨骼权重、裁剪状态和后续光栅化结果并不相同。

可用下列伪代码表示这一特征：

```text
for each fragment in parallel:
    input = interpolate(vertex_attributes, fragment.position)

    if fragment.depth >= depth_buffer[fragment.position]:
        discard fragment

    base_color = sample(texture, input.uv)
    normal = normalize(input.normal)

    if material.has_normal_map:
        normal = decode_normal(sample(normal_map, input.uv))

    lighting = compute_lighting(normal, input.world_position)

    if shadow_map_test(input.world_position) == in_shadow:
        lighting = lighting * shadow_factor

    color = base_color * lighting

    if material.alpha < 1.0:
        framebuffer[fragment.position] =
            blend(framebuffer[fragment.position], color, material.alpha)
    else:
        framebuffer[fragment.position] = color
```

在该伪代码中，片元共享同一程序结构，适合以 SIMT 方式批量执行；但每个片元的 `position`、`uv`、`normal`、`depth` 和 `material` 不同，导致部分 lane 被丢弃，部分 lane 执行 normal map、shadow 或 alpha blending 分支。GPU 必须支持这种分支发散，并通过更多常驻 warp 隐藏分支和访存带来的空闲。

因此，“相似但不完全相同”指的是：大量元素适合进入同一可编程流水线批处理，但硬件必须允许元素携带不同数据，并容忍一定程度的不规则控制流和不规则访存。

这类工作负载具有以下性质：

- 并行元素数量极大。
- 单个元素的控制流和访存路径可能不完全规则。
- 访存和特殊函数延迟必须被隐藏，否则难以满足帧时间约束。
- 可编程性重要，因为 shader 需要表达多样化渲染效果。

因此，GPU 的合理设计目标是提供大量轻量线程，并由硬件在 ready 线程组之间快速切换，以隐藏内存和执行延迟。

### 4.2 从图形渲染推导架构需求

实时渲染要求在固定时间内完成大量相似但不完全一致的小程序实例。若某个实例等待内存，硬件不应空转，而应调度其他 ready 实例继续执行。

由此可推导出 GPU 的基本机制：

1. 将程序组织为大量线程。
2. 将线程分组为 warp，以 SIMT 方式执行。
3. 在每个 SM 中保持多个 warp 常驻。
4. 由 warp scheduler 每周期选择 ready warp 发射。
5. 通过多 warp 并发隐藏访存、特殊函数和流水线延迟。

GPU 的关键不在于单个线程低延迟完成，而在于系统中持续存在可发射的 warp。

### 4.3 SIMT 的意义

SIMT 可理解为“以线程形式编程，以向量方式执行”。程序员使用线程抽象表达并行性；硬件将一组线程组成 warp，并让多个 lane 在同一条指令下并行执行。

SIMT 具有两个直接后果：

- 对规则数据并行程序，SIMT 能高效利用硬件 lane。
- 对分支发散程序，同一 warp 内不同 lane 需分路径执行，有效利用率下降。

因此，GPU 不是通过消除不规则性获得吞吐，而是通过大量线程、warp 调度和存储层次来容忍不规则性。

### 4.4 存储层次与延迟隐藏

GPU 也依赖数据复用，但采用更通用的机制：

- Global memory 提供大容量和高带宽。
- L2 与各级 cache 缓解重复访存。
- Shared memory 为程序员提供显式片上缓存。
- 大容量 register file 支撑大量常驻线程。
- Warp scheduler 用 ready warp 覆盖等待周期。

与 NPU 相比，GPU 不要求所有数据搬运都显式组织为 tile pipeline。程序可结合 cache、coalescing、shared memory 和 warp 调度提高有效带宽。

这种设计扩大了程序覆盖面，可支持图形、科学计算、物理仿真、传统 GPGPU 算法和 AI 算子。但对于高度规则的矩阵乘，如果只依赖标量/SIMT 执行，能效通常不如专用张量阵列。

### 4.5 Tensor Core 的位置

现代 NVIDIA GPU 引入 Tensor Core，是因为 AI 工作负载的重要性足以影响 GPU 内部资源配置。Tensor Core 可视为 GPU 在通用 SIMT 框架内加入的专用矩阵执行单元。

这说明 GPU 并不排斥专用化，而是在通用吞吐处理器框架内吸收专用执行部件：

```text
通用线程并行框架 + 专用矩阵执行单元 + 硬件调度和缓存体系
```

因此，现代 GPU 与 NPU 在 AI 计算上存在一定收敛：二者都需要矩阵阵列和片上复用。但外层执行抽象仍不同。GPU 以 thread、warp 和 block 为基本抽象；NPU 更强调 tile、buffer、MTE 和 event。

### 4.6 设计思想总结

NVIDIA GPU 的体系结构思想可概括为：

```text
用大量线程暴露并行性；
用 SIMT 批量执行相同指令；
用硬件 warp 调度隐藏延迟；
用 cache、shared memory 和 register file 支撑通用吞吐。
```

它优先优化广泛并行工作负载的吞吐和可编程性，而不是只针对单一数据流形态获得最高能效。

## 5. 两种设计路线的对照

| 维度 | 昇腾 NPU | NVIDIA GPU |
|---|---|---|
| 初始目标 | AI 张量计算加速 | 实时图形渲染，后扩展到通用并行计算 |
| 基本抽象 | tile、buffer、dataflow、event | thread、warp、block、SM |
| 主要执行单元 | Cube、Vector、Scalar、MTE 等异构单元 | CUDA core、SFU、Tensor Core、load/store 等 |
| 并行方式 | tile 级数据流并行和矩阵阵列并行 | 大量线程和 warp 级 SIMT 并行 |
| 访存思想 | 软件/编译器显式管理片上存储和搬运 | cache、coalescing、shared memory、warp 调度共同隐藏延迟 |
| 同步方式 | event/token、queue、显式 producer/consumer 边界 | block/warp 同步、memory ordering、调度器 ready 选择 |
| 优势 | 规则张量计算能效高，数据流确定 | 通用性强，可容忍复杂控制流和访存模式 |
| 代价 | 编程和编译器调度复杂，通用性较弱 | 控制和调度硬件复杂，规则矩阵计算需专用单元提高能效 |

## 6. 结论

若目标工作负载是高度规则的张量计算，合理架构倾向于：

```text
固定 tile 计算阵列 + 显式片上存储 + 显式数据搬运 + event/token 同步
```

这对应昇腾 NPU 的设计方向。

若目标工作负载是大量相似但包含分支和不规则访存的并行程序，合理架构倾向于：

```text
大量轻量线程 + SIMT 执行 + 多 warp 常驻 + 硬件调度隐藏延迟
```

这对应 NVIDIA GPU 的设计方向。

二者的差异不是简单的“专用”和“通用”之分，而是计算、存储和调度责任划分不同：

- NPU 将更多调度责任交给编译器和显式数据流，以换取规则张量计算的能效和确定性。
- GPU 将更多调度责任放入硬件线程调度和存储层次，以换取更大的程序覆盖面和延迟容忍能力。

## 7. 学习体系结构的启示

分析这两类芯片时，应从目标函数出发，而不是从模块名称出发：

1. 工作负载是否规则。
2. 数据是否可静态分块和复用。
3. 延迟主要由显式流水隐藏，还是由大量线程隐藏。
4. 编译器、程序员和硬件各自承担多少调度责任。
5. 峰值算力能否被数据通路持续供给。

昇腾 NPU 和 NVIDIA GPU 代表了两种成熟的高吞吐处理器思想。NPU 是以数据流和张量 tile 为中心的专用化路线；GPU 是以线程并行和硬件调度为中心的通用吞吐路线。现代 AI 计算使二者在矩阵执行单元上趋近，但它们关于数据管理、延迟隐藏和调度复杂性分配的基本答案仍然不同。
